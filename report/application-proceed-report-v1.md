# 물품관리 애플리케이션 개발 진행 보고서 (v1)

본 보고서는 제공된 데이터 구조(`init.sql`)를 바탕으로 한 'SeuStock' 물품관리 앱의 개발 순서, 상세 내용 및 예상 기간을 정리한 리포트입니다.

---

## 1. 프로젝트 개요
- **프로젝트 명**: SeuStock (물품 및 재고 관리 시스템)
- **주요 목적**: 사용자가 공간(Space), 선반(Shelf), 박스(Box)로 구성된 계층적 위치에 품목(Item)의 재고(Stock)를 등록하고 관리하며, 입출고 이력(Transaction)을 추적할 수 있는 시스템 구축
- **기술 스택**: Spring Boot, MyBatis, PostgreSQL, Thymeleaf (또는 REST API + 프론트엔드 프레임워크)

---

## 2. 데이터 구조 요약
제공된 SQL 설계를 바탕으로 한 핵심 엔티티 구조는 다음과 같습니다:
- **사용자(Users)**: 시스템 이용 주체
- **위치 체계**: 공간(Spaces) > 선반(Shelves) > 박스(Boxes)의 3단계 계층 구조
- **품목(Items)**: 관리할 물품의 마스터 정보 (이름, 설명 등)
- **재고(Stocks)**: 특정 품목이 어느 위치에 얼마나 있는지에 대한 실물 정보
- **이력(Stock Transactions)**: 입고, 출고 등 재고 수량의 변화 기록

---

## 3. 개발 로드맵 및 상세 단계

### Phase 1: 기반 인프라 및 사용자 관리 (1주)
- **인프라 설정**: Docker Compose를 이용한 PostgreSQL 환경 구축 (완료 단계)
- **사용자 관리 (Users)**:
    - 회원가입 및 로그인 기능 구현
    - Spring Security 기반의 인증/인가 처리
- **기본 레이아웃 구성**: 공통 네비게이션 및 대시보드 UI 프레임워크 구축

### Phase 2: 위치 체계 및 마스터 정보 관리 (1.5주)
- **위치 관리 (Spaces, Shelves, Boxes)**:
    - 공간 생성/조회/수정/삭제 (CRUD)
    - 선반 및 박스 등록 (계층 구조 반영)
- **품목 관리 (Items)**:
    - 물품 마스터 정보 등록 및 관리
    - 품목별 목록 조회 및 검색 기능

### Phase 3: 핵심 재고 관리 및 트랜잭션 (1.5주)
- **재고 관리 (Stocks)**:
    - 특정 위치(공간/선반/박스)에 물품 재고 할당
    - 위치별/품목별 현재고 현황 조회
- **입출고 처리 (Transactions)**:
    - 재고 증가(입고) 및 감소(출고) 기능
    - 트랜잭션 기록 자동 생성 (History)
    - 재고 변동 이력 리스트 조회

### Phase 4: 대시보드 및 고도화 (1주)
- **대시보드**: 전체 재고 현황 요약, 최근 입출고 내역 시각화
- **검색 및 필터링**: 위치별, 품목별 검색 기능 강화
- **예외 처리 및 최적화**: 트랜잭션 정합성 체크, 데이터 검증 강화

---

## 4. 상세 개발 내역 (Back-end & Front-end)

| 구분 | 주요 개발 내용 | 상세 비고 |
| :--- | :--- | :--- |
| **API** | RESTful API 설계 | `/api/spaces`, `/api/items`, `/api/stocks` 등 |
| **Business Logic** | 재고 계산 로직 | 입출고 시 `stocks.quantity` 업데이트 및 `transactions` 기록 보장 |
| **Validation** | 데이터 무결성 검증 | 박스 존재 시 상위 선반 필수 체크 등 (SQL CHECK 제약 연동) |
| **UI/UX** | 반응형 웹 디자인 | 모바일에서도 위치 확인 및 재고 입력이 가능하도록 설계 |

---

## 5. 예상 소요 기간 요약
- **총 예상 기간**: 약 5주
- **일정 배분**:
    - **1주차**: 프로젝트 셋업 및 사용자 인증 완료
    - **2~3주차**: 위치 및 품목 관리 API & 화면 완성
    - **4주차**: 핵심 재고 변동 로직 및 이력 조회 구현
    - **5주차**: 통합 테스트, 버그 수정 및 최종 배포 준비

---

## 6. 결론 및 제언
현재 데이터 구조가 계층적으로 잘 설계되어 있어(Space-Shelf-Box), 이를 UI에서 트리 구조나 브레드크럼(Breadcrumb) 형태로 직관적으로 보여주는 것이 중요합니다. 특히 `stocks` 테이블에서 위치 정보가 유연하게 설계되어 있으므로, 위치가 지정되지 않은 재고(미분류)에 대한 처리 프로세스를 추가하면 더욱 견고한 앱이 될 것입니다.
