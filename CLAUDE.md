## Commands

```bash
# Run the application (auto-starts Docker Compose for PostgreSQL)
./gradlew bootRun

# Build
./gradlew build

# Run tests (uses H2 in-memory DB, no Docker needed)
./gradlew test

# Run a single test class
./gradlew test --tests "com.seu.seustock.SeuStockApplicationTests"
```

The `spring-boot-docker-compose` dependency automatically starts `compose.yaml` when the app runs via Gradle. Docker must be running. PostgreSQL is exposed on port **5433** (mapped from container's 5432).

## Architecture

Spring Boot 4.0.6 / Java 25 / Gradle (Kotlin DSL) web app for personal inventory management.

**Request flow:** HTTP → `@Controller` → MyBatis `Mapper` interface → XML mapper → PostgreSQL

**Key conventions:**
- DTOs live in `com.seu.seustock.model.dto` and are mapped via `mybatis.configuration.map-underscore-to-camel-case=true` — no manual column aliasing needed unless a `<resultMap>` is already defined.
- MyBatis SQL lives in XML files under `src/main/resources/mapper/**/*.xml`. The mapper namespace must match the fully-qualified Java interface name (e.g., `com.seu.seustock.mapper.UserMapper`).
- Database rows use two IDs: an internal `id` (SERIAL) for FK joins, and a public `external_id` (UUID, auto-generated by Postgres) for any external-facing references. Never expose the internal `id` in APIs or URLs.
- Spring Security is present as a commented-out dependency — not yet wired up.
- Frontend uses Thymeleaf templates + HTMX (`io.github.wimdeblauwe:htmx-spring-boot-thymeleaf`).

## Database schema

Source of truth: `schema/schema-v1.sql`. The file `docker/postgres/init/init.sql` is a copy used for Docker container initialization — keep them in sync when making schema changes.

Domain hierarchy (physical storage):
```
users → spaces → shelves → boxes
```

Inventory model:
- `items`: master catalog of item definitions (no location or quantity info)
- `stocks`: links an `item` to a location (`space` required; `shelf` and `box` optional) with a `quantity`
- `stock_transactions`: append-only ledger of quantity changes (`quantity_delta`) against a `stock` row

The `stocks.quantity` has a `CHECK (quantity >= 0)` constraint. Stock mutations must go through `stock_transactions` to maintain an audit trail.

Reference queries for each entity are kept in the `query/` directory.
